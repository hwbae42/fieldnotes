# 에이전트 메모리 / 컨텍스트 관리 — 개요

## 왜 이 주제인가

2025–2026 사이에 LLM의 컨텍스트 윈도우는 200K, 1M, 심지어 2M 토큰까지 늘어났다. 그래서 "컨텍스트 관리는 끝난 문제 아닌가?"라는 의문이 자주 나온다. 실제 프로덕션 에이전트를 운영해 보면 답은 정반대다. 컨텍스트가 커진 만큼 **무엇을 넣을지 결정하는 일**의 비용이 더 비싸졌다.

Anthropic은 이 흐름을 가리켜 **context engineering** 이라고 부른다(2025년 가을 엔지니어링 블로그). 정의는 단순하다 — "한정된 컨텍스트 윈도우에 LLM 추론에 필요한 *최적의 토큰 집합*을 큐레이션하는 일." 프롬프트 엔지니어링이 "어떻게 쓰는가"였다면, context engineering은 "무엇을 넣는가" 그 자체가 1급 시스템 설계 문제로 올라온 것이다.

## Long-context 모델로도 안 풀리는 부분

긴 컨텍스트 윈도우가 등장한 뒤에도 다음 문제는 그대로 남는다.

### 1. Context rot (컨텍스트 부패)

트랜스포머 어텐션은 토큰 수에 대해 n² 관계를 가진다. 입력 토큰이 길어질수록 모델이 분배할 수 있는 **어텐션 예산**은 토큰당 더 작아지고, 노이즈 토큰은 신호 토큰의 가중치를 빨아먹는다. 200K 윈도우에 200K를 채우는 것보다 50K만 잘 큐레이션해 넣는 편이 거의 항상 더 좋다는 보고가 누적되고 있다.

### 2. Context anxiety / mode collapse

Cognition이 Devin을 Sonnet 4.5로 옮기면서 발견한 현상은 흥미롭다 — 모델이 자기 컨텍스트를 *의식*하기 시작하면, 한계가 가까워질수록 작업을 너무 빨리 마무리하려 하고("아무거나 일단 결정", "있는 정보로 close out"), 자기 상태를 부실하게 요약한다. 이는 long-context 자체가 아니라 *모델이 자기 한계를 인지하는 능력*이 만든 부작용이다. Cognition은 1M 베타를 켜고 200K에서 캡을 거는 식의 우회까지 동원했다.

### 3. 비용과 지연

200K 토큰을 매 턴 통째로 흘려 넣으면 토큰 비용·캐시 미스·첫 토큰 지연이 모두 비례로 커진다. Mem0가 LOCOMO에서 보여준 숫자는 직관적이다.

| 접근 | 정확도 | P95 지연 | 토큰/대화 |
|---|---|---|---|
| Full-context | 72.9% | 17.12s | ~26,000 |
| Mem0g (graph) | 68.4% | 2.59s | ~1,800 |
| Mem0 (vector) | 66.9% | 1.44s | ~1,800 |
| RAG | 61.0% | 0.70s | — |
| OpenAI Memory | 52.9% | — | — |

정확도 4–6%p를 양보하는 대신 지연은 6–10배, 토큰은 14배 줄어든다. 프로덕션에서 이 트레이드는 거의 항상 선택적 메모리 쪽이 이긴다.

### 4. 세션을 가로지르는 상태

긴 컨텍스트는 "한 세션 안에서" 더 많이 기억할 뿐이다. 세션 N에서 학습한 사용자 선호를 세션 N+1에 전달하려면 외부 저장소가 필요하다. 컨텍스트 윈도우는 **워킹 메모리**일 뿐, 영속 메모리가 아니다.

### 5. 멀티 에이전트의 컨텍스트 격리

Anthropic의 멀티 에이전트 리서치 시스템은 lead agent가 단일 컨텍스트로 모든 서브태스크를 다루지 않고, 서브에이전트들에게 격리된 컨텍스트 윈도우를 주고 결과 요약만 받는다. 이때 컨텍스트는 의도적으로 "조각조각" 분리된다.

## 컨텍스트 윈도우 ≠ 메모리

이 두 개념은 자주 섞여 쓰이지만 다른 것이다.

| 항목 | 컨텍스트 윈도우 | 메모리 |
|---|---|---|
| 수명 | 한 세션 / 한 호출 | 세션을 가로질러 영속 |
| 쓰기 | 매 턴 모델이 받는 것 | 명시적 도구 호출 / 백그라운드 추출 |
| 업데이트 | 추가만 가능 (그리고 압축/삭제) | CRUD 모두 가능 |
| 검색 | 위치 기반 (앞뒤 토큰) | 시맨틱·키워드·시간·관계 기반 |
| 비용 모델 | 토큰당 비례 | 저장·인덱싱 비용 + 검색 비용 |
| 적절한 비유 | RAM | 디스크 + 캐시 계층 |

MemGPT의 통찰이 정확하다 — LLM을 운영체제처럼 다루면, 컨텍스트는 RAM이고 외부 저장소는 디스크다. 둘 사이를 옮겨 다니는 정책이 바로 메모리 시스템이다.

## 메모리 분류

학술 문헌(특히 CoALA, "Memory for Autonomous LLM Agents" 서베이)과 LangChain·Letta 같은 프레임워크가 정착시킨 분류는 다음과 같다.

### 시간 축으로

- **Working memory (단기)** — 현재 컨텍스트 윈도우에 들어 있는 것. 스크래치패드, to-do, 중간 추론.
- **Long-term memory (장기)** — 외부 저장소에 영속화. 세션을 가로질러 살아남는다.

### 내용 종류로 (인지심리학에서 차용)

- **Semantic memory (의미 기억)** — 사실·지식. "사용자는 Python보다 TypeScript를 선호한다", "API 핸들러는 `src/api/handlers/`에 있다." 일반화 가능, 시간에 둔감.
- **Episodic memory (일화 기억)** — 특정 시점의 사건. "지난 화요일 X 접근법을 시도했고 Z 때문에 실패했다." 타임스탬프가 핵심.
- **Procedural memory (절차 기억)** — 어떻게 하는가. 시스템 프롬프트 규칙, few-shot 예시, "커밋 전 `npm test`를 돌려라" 같은 워크플로.

LangChain은 procedural을 "거의 아무도 자동 업데이트하지 않는 영역"이라고 솔직하게 말한다 — 모델 가중치를 학습하는 게 아니라 시스템 프롬프트를 일부 갱신하는 정도가 현실이다.

### 쓰기 정책으로

- **Hot path** — 응답하기 전에 에이전트가 명시적으로 메모리를 갱신한다 (ChatGPT memory 방식). 반응 즉시 반영되지만 응답 지연이 늘어난다.
- **Background extraction** — 별도 프로세스가 대화가 끝난 뒤 비동기로 추출·저장 (Mem0 v1.0+ 기본). 응답에 지연을 주지 않지만, 다음 턴부터는 못 쓰일 수 있다.
- **User feedback** — 좋아요/싫어요로 가중치를 조정 (에피소딕 인출 품질에 직결).

## Context engineering의 4가지 큰 동작

LangChain이 정리한 Write / Select / Compress / Isolate 분류는 의외로 잘 들어맞는다.

```
                  ┌─────────────────────────────┐
                  │   Context Window (LLM)      │
                  └──────────────┬──────────────┘
                                 ▲
                                 │ select
                                 │ (RAG, memory retrieval, tool desc)
                                 │
       ┌─────────────────────────┼─────────────────────────┐
       │                         │                         │
   ┌───┴──────┐        ┌─────────┴──────────┐      ┌──────┴────────┐
   │  System  │        │  Long-term Memory  │      │  Tool Results │
   │  Prompt  │        │ (vector / graph /  │      │  (just in time│
   │ + Skills │        │  files / SQLite)   │      │   retrieval)  │
   └──────────┘        └─────────┬──────────┘      └───────────────┘
                                 ▲
                                 │ write (extract, summarize)
                                 │ + manage (compress, decay, conflict)
                                 │
                  ┌──────────────┴──────────────┐
                  │   User Input + Past Turns   │
                  └─────────────────────────────┘

                  Isolate: subagents get fresh context windows;
                           parent only sees scoped summary.
                  Compress: auto-compaction, hierarchical summary,
                           context editing (clear stale tool calls).
```

- **Write** — 컨텍스트를 외부에 적기 (스크래치패드, MEMORY.md, 벡터 DB).
- **Select** — 다음 호출에 무엇을 끌어올 것인가 (RAG, 메모리 인출, 도구 설명 RAG).
- **Compress** — 토큰 줄이기 (auto-compaction, 계층 요약, context editing).
- **Isolate** — 컨텍스트를 쪼개기 (서브에이전트, sandbox, state schema).

이 4개는 동시에 쓰는 도구이지 배타적 선택이 아니다. 한 시스템 안에서 4개가 다 일어난다.

## "Context engineering"이라는 흐름이 등장한 배경

2024년까지는 "프롬프트 엔지니어링" 한 단어로 대부분이 설명됐다. 2025년 들어 LangChain, Anthropic, Cognition이 거의 동시에 "이건 프롬프트 엔지니어링이 아니라 더 큰 무엇"이라고 말하기 시작했다. 이유는 단순하다.

1. **에이전트의 등장** — 1회 호출이 아니라 수백 턴, 수십 도구 호출, 멀티 세션을 거치는 워크플로가 표준이 됨. 매 턴마다 컨텍스트를 *다시 설계*해야 한다.
2. **프롬프트 외 입력의 폭증** — 도구 결과, 검색 결과, 메모리 인출, 멀티모달 입력이 시스템 프롬프트보다 훨씬 큰 영역을 차지한다. "프롬프트만 잘 쓰자"는 게 의미가 없어졌다.
3. **모델의 자기 컨텍스트 인식** — Sonnet 4.5부터 모델이 자기 토큰 잔량을 추정한다. 이는 외부에서 "어떻게 보이게 만들지"가 출력 품질에 직접 영향을 준다는 뜻.

따라서 context engineering은 마케팅 용어이자 동시에 진짜 새로운 엔지니어링 분야다 — 프롬프트, 도구, 메모리, 검색, 압축, 격리를 한 시스템으로 다루는 일.

## 메모리 백엔드의 큰 그림

| 백엔드 | 강점 | 약점 | 적합한 곳 |
|---|---|---|---|
| **파일 시스템** (Claude Code MEMORY.md, Memory tool) | 투명함, grep 가능, 사람이 직접 편집·감사 | 시맨틱 검색 안 됨, 큰 코퍼스에 불리 | 코딩 에이전트, 프로젝트 단위 노트 |
| **SQLite / 관계형** | 구조화, 트랜잭션, 빠른 키 조회 | 자유 텍스트 검색 빈약 | 사실(facts) 저장, 사용자 프로필 |
| **벡터 DB** (Pinecone, Qdrant, pgvector) | 시맨틱 유사도, 대규모 코퍼스 | 다중 홉 추론 불가, 시간 추론 약함 | 대화 이력, 문서 RAG, 에피소드 |
| **지식 그래프** (Neo4j, Kuzu, Graphiti) | 관계 추론, temporal validity, 설명 가능 | 온톨로지 설계 부담, 콜드스타트 비용 | 의료/법률처럼 관계가 중요한 도메인 |
| **Hybrid (Graphiti, Zep, GraphRAG)** | 시맨틱 + 관계 + 시간을 함께 | 복잡도, 인프라 다중화 | 엔터프라이즈, 충돌 해소가 필요한 곳 |

2026년 시점의 컨센서스(Mem0 State of AI Agent Memory 보고서, Atlan 비교): **벡터 + 그래프 하이브리드가 엔터프라이즈 표준**이다. 단, 작은 프로젝트는 파일 시스템 / SQLite로도 충분히 가는 경우가 많다.

## RAG와 에이전트 메모리의 관계

가장 흔한 혼동. 둘 다 "외부에서 정보 가져와 컨텍스트에 넣기"라 비슷해 보이지만 본질이 다르다.

| | RAG | Agent Memory |
|---|---|---|
| 동작 | Read-only | Read-write |
| 출처 | 정적 코퍼스 (문서) | 동적 (대화·관찰·반영에서 추출) |
| 갱신 | 오프라인 인덱싱 | 인라인 또는 백그라운드 |
| 시간 | 시간 무관 | 타임스탬프, 신선도, 만료 |
| 충돌 | 새 청크 추가 (충돌 안 풀림) | 명시적 conflict resolution |
| 적합 | "에펠탑 높이는?" 같은 사실 | "사용자가 작년에 Python에서 TS로 옮겼다" |

LeonieMonigatti의 비유가 유용하다 — RAG는 "도서관에서 책 빌리기", agent memory는 "노트 쓰면서 대화하기." 실제 시스템에서는 둘을 함께 쓴다 (semantic 메모리는 RAG처럼, episodic은 메모리 시스템으로).

## 이 문서에서 다루는 것 / 다루지 않는 것

**다룬다:**
- 에이전트가 한 세션 내, 그리고 세션을 가로질러 컨텍스트를 어떻게 관리하나
- 압축·요약·메모리 백엔드의 트레이드오프
- 2025–2026의 실제 구현 (Anthropic memory tool, Claude Code, Mem0, Letta, Zep)
- 멀티 에이전트에서의 컨텍스트 격리

**다루지 않는다:**
- 모델 가중치에 새 정보를 학습시키는 기법 (continual learning, LoRA 기반 personalization)
- RAG 검색 품질 자체의 깊은 튜닝 (chunking 전략, re-ranking 모델)
- 음성 / 멀티모달 컨텍스트 특수성
